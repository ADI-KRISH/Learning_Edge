# Chapter 1: Introduction to Databases and the Relational Model
A database is an organized collection of structured data managed by a Database Management System (DBMS). The relational model, introduced by E.F. Codd in 1970, organizes data into tables (relations) consisting of rows (tuples) and columns (attributes). Each table has a primary key — a column or set of columns that uniquely identifies each row. A foreign key in one table references the primary key of another, establishing relationships between tables. The three types of relationships are one-to-one, one-to-many, and many-to-many (the last requires a junction table). A schema defines the structure of the database: the tables, their columns, data types, and constraints. Constraints enforce data integrity: NOT NULL (a field cannot be empty), UNIQUE (no duplicate values in a column), CHECK (values must satisfy a condition), and FOREIGN KEY (referential integrity between tables). The ACID properties — Atomicity, Consistency, Isolation, and Durability — guarantee reliable transaction processing in a DBMS.

# Chapter 2: SQL — Querying and Manipulating Data
SQL (Structured Query Language) is the standard language for interacting with relational databases. The SELECT statement retrieves data: SELECT columns FROM table WHERE condition. JOINs combine rows from two or more tables: INNER JOIN returns only matching rows; LEFT JOIN returns all rows from the left table with NULLs for non-matching right rows; FULL OUTER JOIN returns all rows from both tables. GROUP BY aggregates rows sharing a common value, used with aggregate functions like COUNT(), SUM(), AVG(), MIN(), and MAX(). HAVING filters groups (like WHERE but applied after aggregation). Subqueries are nested SELECT statements used within WHERE or FROM clauses. The ORDER BY clause sorts results; LIMIT restricts the number of rows returned. Data Manipulation Language (DML) includes INSERT (add rows), UPDATE (modify rows), and DELETE (remove rows). Data Definition Language (DDL) includes CREATE TABLE, ALTER TABLE, and DROP TABLE. Indexes (typically B-trees) dramatically speed up query performance on large tables by avoiding full table scans.

# Chapter 3: Database Normalization and Design
Normalization is the process of organizing a database to reduce data redundancy and improve integrity. The normal forms provide progressive levels of normalization. First Normal Form (1NF) requires that all column values are atomic (indivisible) and there are no repeating groups. Second Normal Form (2NF) requires 1NF and that every non-key attribute is fully functionally dependent on the entire primary key (no partial dependencies). Third Normal Form (3NF) requires 2NF and that no non-key attribute is transitively dependent on the primary key. Boyce-Codd Normal Form (BCNF) is a stricter version of 3NF where every determinant must be a candidate key. Denormalization deliberately introduces redundancy to improve read performance in analytical workloads. Entity-Relationship (ER) diagrams are used during the design phase to model entities, attributes, and relationships visually before translating them into a relational schema.

# Chapter 4: Transactions, Concurrency, and NoSQL Databases
A transaction is a sequence of operations treated as a single logical unit. The ACID properties ensure correctness: Atomicity (all-or-nothing execution), Consistency (database moves from one valid state to another), Isolation (concurrent transactions do not interfere), and Durability (committed transactions survive failures). Concurrency control prevents anomalies like dirty reads (reading uncommitted data), non-repeatable reads (same query returns different values), and phantom reads (new rows appear between reads). Isolation levels in SQL range from Read Uncommitted to Serializable. Two-Phase Locking (2PL) acquires all locks before releasing any, guaranteeing serializability. Deadlocks occur when transactions wait for each other's locks and are resolved by timeout or detection algorithms. NoSQL databases abandon strict relational structure for scalability: Document stores (MongoDB) store JSON-like documents; Key-Value stores (Redis) offer ultra-fast lookups; Column-family stores (Cassandra) optimize for wide-row queries; Graph databases (Neo4j) model connected data as nodes and edges. NoSQL systems often trade ACID for BASE (Basically Available, Soft-state, Eventually consistent).
